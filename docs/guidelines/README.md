# 台灣電商輔助工具 App

## 專案簡介
這是一個專為 **Shopline / Cyberbiz 賣家** 設計的「貼身電商助手」。  
核心精神：幫助賣家「先設定目標 → 系統自動拆解 → 提供具體行動方案」，並透過遊戲化機制，讓使用工具變得更有趣、更持續。  

本專案同時結合：
- **Vibe Coding**：工程開發的任務拆解、模組化架構、清單管理。  
- **Vibe Marketing**：商業目標導向、八角框架驅動、遊戲化陪跑。  

---

## 使用情境
- 九月要達成 **100 萬業績**  
  → 工具自動拆解：需要多少流量 / 轉換率 / 客單價，並對照現況數據。  
- 系統幫助判斷電商類型：  
  - **爆品型**：主打單品、行銷快速拉量  
  - **通路型**：SKU 多，注重轉換與會員經營  
  - **FMCG 高頻消費**：靠組合包、回購提升營收  
- 根據類型 + 缺口 → 提供「行銷活動卡片」與「操作建議」。

| 驅動力 | 功能方向 | 賣家體驗 | KPI |

## 功能模組
### 1. 目標設定與拆解
- 輸入營收目標（例：100 萬）  
- 系統拆解 → 流量 / 轉換率 / 客單價需求  
- 缺口分析 → 顯示目前差多少  

### 2. 活動助手
- 根據缺口，推薦三類方案：  
  - **流量型**（廣告投放、導流活動）  
  - **轉換型**（優惠券、免運、會員回購）  
  - **客單價型**（組合包、滿額贈）  
- 以「活動卡片」方式呈現，方便挑選  

### 3. 遊戲化成長
- **任務系統**：每日 / 每週可完成的簡單步驟  
- **徽章 / 等級**：完成任務 → 解鎖成長路線  
- **排行榜**：與其他賣家比較營運進度  
- **驚喜點數**：完成特定成就 → 兌換工具功能或模版  

### 4. 數據同步
- 支援 Shopline / Cyberbiz API  
- 自動帶入：流量、轉換率、客單價、營收  
- 提供即時進度條 →「距離目標還差 XX%」

## 技術架構
### 前端
- React + Tailwind  
- 元件：目標看板、活動卡片、任務中心  

### 後端
- Node.js (NestJS/Express)  
- API 模組：目標、活動、點數、通知  
- WebSocket：即時進度推播  

### 資料庫
- PostgreSQL：訂單、用戶、活動

- MongoDB：UGC / 任務紀錄  

### DevOps
- Docker 容器化  
- GitHub Actions CI/CD  
- 兩週迭代

## 商業目標對齊 (Business Goals Mapping)
每個功能需明確標註以下資訊：
- **Business Goal**：解決的商業目標  
- **Octalysis Driver**：對應的八角框架驅動  
- **KPI Link**：影響的數據指標  
- **Gamification Element**：內建的遊戲化互動  

### 範例
```markdown
# 功能：快閃活動生成器

## Business Goal
提高短期轉換率 (CVR)，加速庫存周轉。

## Octalysis Driver
- 稀缺與焦慮（Scarcity & Impatience）
- 進步與成就（任務完成 → 勳章）

## KPI Link
- 活動 CVR
- 售罄時間

## Gamification Element
- 倒數條 UI
- 完成快閃設定 → 解鎖「行銷小將」徽章


## 🕹️ 遊戲規則（使用介面設計原則）

1. **Portal 為核心入口**  
   - 每個網頁都應承載明確功能（或多個相關功能）。  
   - 使用者流程必須由 Portal 頁面引導，逐步完成任務與操作。  

2. **避免無目的頁面**  
   - 不要隨意新增網頁。  
   - 新增頁面必須有清楚的目的與使用情境，並對應到使用者目標或八角框架驅動力。  

3. **元素重用原則**  
   - 現有元素（例如進度條、徽章、缺口分析卡片、活動推薦卡片）是核心互動元件，應該保留並擴充，而非替換。  
   - Demo 頁面中的元素（如：進度條顯示完成度、徽章展示、任務卡片）需轉化為「可操作」或「可追蹤」的正式功能。  

4. **引導性設計**  
   - 所有互動需有「下一步」提示，避免使用者迷路。  
   - 範例：完成目標設定 → 自動進入缺口分析 → 提示挑選活動方案。  

5. **遊戲化一致性**  
   - 任務、徽章、驚喜獎勵需全程貫穿，保持持續性動力。  
   - 每個操作必須對應到「八角框架」之一，並清楚標示該操作的獎勵或進展。  

6. **簡單即上手**  
   - 頁面設計需降低學習成本，保持單一頁面完成一組任務的邏輯。  
   - 範例：在同一頁面即可輸入目標、檢視缺口、選擇活動，而不是跳轉多個分頁。  

7. **Demo vs. Production 區隔**  
   - Demo 頁面僅作展示，不可直接拿來當正式功能。  
   - 正式版本必須移除假數據與硬編排，改用後端 API 或動態數據來源。
